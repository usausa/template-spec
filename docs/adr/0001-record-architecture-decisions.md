## 背景 / 課題
AI 駆動開発では、生成物は「もっともらしいが理由が残らない」性質を持つ。決定の "なぜ" を残さないと、
乖離が **意図した逸脱** なのか **ただのドリフト** なのか判別できず、後からドキュメントを安全に直せない。

## 決定
設計上の決定(Why)は `docs/adr/` に **ADR として追記** する。過去 ADR は編集せず、覆すときは新 ADR を起こす。
現状仕様(What/How)は ADR に書かない(それはコード+テスト/生成物の領域)。

## 検討した代替案
- **案A: ADR を持たない** / 却下理由: 決定の背景が失われ、保守で "なぜ" を追えなくなる。生成物は "もっともらしいが理由が残らない" ため、放置すると意図とドリフトの区別がつかなくなる。
- **案B: 設計書(DES)に決定も混ぜる** / 却下理由: 設計書は更新され上書きされる。決定は不変・追記で残すべき性質が違う。

## 結果(トレードオフ・影響)
- 決定のたびに ADR を書く小さなコストが増える代わりに、意図とドリフトを区別でき、ドキュメントを延命できる。
- `/adr` コマンドと `write-adr` スキルで作成を支援。`/trace` で「決定漏れ」を検査する。
